
     REGISTRATION FLOW VERIFICATION - Students vs Mentors


 VERIFICATION SUMMARY:
The current Register.js implements IDENTICAL flow for both Student and Mentor
registrations with only one difference: Contact Visibility setting (Mentors only)


 REGISTRATION FLOW (Both Roles - Lines 1-625)


STEP 1: ROLE SELECTION (Lines 88-92)
 Student  Click 'Student' card
 Mentor  Click 'Mentor' card  
 Both proceed to STEP 2: Email Verification

STEP 2: EMAIL VERIFICATION (Lines 94-120)
 Student: Must use @gct.ac.in email (Line 100-105)
 Mentor: Can use ANY valid email (Line 107-111)
 Click 'Send OTP' button
 Backend: POST /api/auth/send-otp
 Email sent with 6-digit OTP
 60-second countdown timer starts

STEP 3: OTP VERIFICATION (Lines 122-142)
 Enter 6-digit OTP code
 Click 'Verify OTP' button
 Backend: POST /api/auth/verify-otp
 On success: emailVerified = true
 Both proceed to STEP 4: Registration Form

STEP 4: REGISTRATION FORM (Lines 144-208)
 Common Fields (Both):
    Full Name (required)
    Password (required, min 6 chars)
    Department (required)
    Email (readonly, already verified)

 Student-Only Fields:
    Graduation Year (required)

 Mentor-Only Fields:
     Company Placed At (required)
     Position (required)
     Placement Year (required)
     Location/Place (required)
     Phone Number (required)
     LinkedIn Profile (required)
     Contact Visibility (required)
         PUBLIC (default) - Everyone can see contact details
         ADMIN_ONLY - Only admins can see contact details

STEP 5: SUBMIT & AUTO-LOGIN
 Backend: POST /api/auth/register
 Response includes user object
 Auto-login via AuthContext
 Student  /student-dashboard
 Mentor  /mentor-dashboard

 BACKEND API ENDPOINTS USED:

 Response: { success: boolean, message: string }

     fullName, email, password, role: 'STUDENT',

     placementYear, location, phoneNumber,
     linkedinProfile, contactVisibility
   }
